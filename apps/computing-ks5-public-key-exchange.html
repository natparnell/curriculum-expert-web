🔑 Public Key Exchange

See how asymmetric encryption protects a message across the internet. Encrypt with a public key, watch only the matching private key unlock it, then try to break it.

KS5 Computing • Networking, the internet and security

Teacher aim: Pupils model RSA-style public-key cryptography. They learn why the public key can be shared freely, why only the private key decrypts, and how digital signatures and tampering checks defend message integrity. Pitched at A-level networking and security content.

1 The key pair

Every party generates a linked pair of keys. The public key is published to the world; the private key never leaves the owner. They are mathematically related but you cannot feasibly work out the private key from the public one.

Mallory cannot read the message, but could she change it in transit? Tamper with the ciphertext, then let Priya decrypt and see the integrity result.

After tampering, run Attempt decryption in Section 3 (with the private key) to see the result. Real systems add a digital signature or MAC so any change is detected.

5 Why both kinds exist

Asymmetric encryption solves key distribution but is slow. In practice (e.g. HTTPS / TLS), it is used to exchange a fast symmetric session key. This is the hybrid model.

🔐🔑 Asymmetric (public-key)

  • Two linked keys: public and private.
  • No need to pre-share a secret: solves key distribution.
  • Enables digital signatures and authentication.
  • Computationally expensive, so slow for large data.
  • Examples: RSA, ECC, Diffie-Hellman exchange.

🔑 Symmetric (shared key)

  • One shared secret key for both encrypt and decrypt.
  • Very fast: ideal for bulk data once a session starts.
  • Problem: how do both sides agree the key securely?
  • Solved by exchanging it under asymmetric encryption.
  • Examples: AES, ChaCha20.
