Searching Algorithms

Teaching aim: understand and compare two key searching algorithms. Linear search checks each item in turn until the target is found or the list ends. Binary search repeatedly halves a sorted list, comparing with the middle item. Watch both run side by side, count the comparisons, and see why binary search is far faster but only works on sorted data.

4 Check your understanding

Why does binary search only work on a sorted list?

Binary search uses the middle item to decide which half to throw away: if the target is bigger than the middle, it must be in the right half; if smaller, the left half. That decision is only safe when the list is in order. On an unsorted list there is no guarantee the target is on the side binary search keeps, so it can discard the half that actually held the answer and report it missing.

Linear search is slower on big lists. So why is it still useful?

Linear search works on any list, sorted or not, and needs no preparation. For a short list, or one that changes constantly so sorting it would be wasted effort, linear search is simple and perfectly fast enough. Binary search only pays off when the list is large and already sorted (or worth sorting once and searching many times).

For a list of 1,000 sorted items, roughly how many comparisons does each search need in the worst case?

Linear search may check all 1,000 items. Binary search halves the list each time: 1000 then 500 then 250 then 125 ... reaching 1 in about 10 comparisons (because 2 to the power 10 is 1024). So binary search needs around 10 comparisons against up to 1,000 for linear search. Each time you double the list, binary search only needs one extra comparison.

What is the "worst case" for linear search?

The worst case is when the target is the very last item, or is not in the list at all. Then linear search has to check every single item before it can answer. Its best case is when the target is the first item, found in one comparison.

Teacher and learner notes: how these algorithms work

Linear search (also called sequential search):

  • Start at the first item.
  • Compare it with the target. If it matches, stop: found.
  • If not, move to the next item and repeat.
  • If you reach the end without a match, the target is not in the list.
  • Worst case: n comparisons for a list of n items.

Binary search (requires a sorted list):

  • Look at the middle item of the part of the list still in range.
  • If it matches the target, stop: found.
  • If the target is smaller, discard the middle and everything to its right, keep the left half.
  • If the target is larger, discard the middle and everything to its left, keep the right half.
  • Repeat on the half that remains until found or no items are left.
  • Worst case: about log₂(n) comparisons, so each doubling of the list adds only one comparison.

The trade-off: binary search is dramatically faster on large lists, but it needs the data to be sorted first. Sorting takes time, so binary search is the clear winner when you sort once and search many times.
