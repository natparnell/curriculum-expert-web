Run-length encoding (RLE) Lossless

RLE replaces sequences of the same value (a "run") with a single value plus a count. It works well when data has long runs, for example simple images with blocks of one colour.

Try AAAABBBCCDAAAA or WWWWWWWWWWWWBWWWWWWWWWWWWBBB. Letters with long runs compress; alternating letters do not.

Colour:

RLE scans the grid row by row (raster order), so horizontal bands of one colour compress best.

Huffman coding Lossless

Huffman coding gives short binary codes to common characters and longer codes to rare ones. Repeatedly combine the two lowest-frequency nodes to build a tree, then read each character's code by walking from the root (left = 0, right = 1).

Classic example: GO GO GOPHERS. Spaces are treated as a character.

Type a message and press Build tree to start. Then press Step to combine the two lowest-frequency nodes one at a time.

Lossy vs lossless: which and when? Explainer

✓ Lossless compression

The original data is reconstructed exactly. Nothing is discarded; redundancy is encoded more efficiently.

  • Methods include RLE and Huffman coding (both on J277).
  • File can be restored bit-for-bit identical to the original.
  • Compression ratio is usually smaller than lossy.
Use when: text, source code, spreadsheets, executables, or any file where losing a single character would corrupt it. Formats: ZIP, PNG, FLAC.

⚠ Lossy compression

Some data is permanently removed, usually detail the human eye or ear is unlikely to notice. The original cannot be recovered.

  • Achieves much smaller files than lossless.
  • Quality drops; repeated re-saving degrades the file.
  • Not used in J277's worked methods, but assessed as a concept.
Use when: photographs, streamed music and video, where small quality loss is acceptable for far smaller files. Formats: JPEG, MP3, MP4.

Key J277 idea: choose lossless when the data must remain identical; choose lossy when reducing file size matters more than perfect fidelity.

Exam-style questions Check understanding

OCR J277-style questions. Attempt each one, then reveal the model answer.

2 marksState the difference between lossy and lossless compression.
Model answer: Lossy compression permanently removes some data so the original cannot be fully recovered, whereas lossless compression encodes the data more efficiently so the original can be reconstructed exactly with no loss.
2 marksGive one suitable use of lossy compression and one of lossless compression. Justify each choice.
Model answer: Lossy is suitable for a photograph or streamed song because a small loss of detail is unnoticeable and gives a much smaller file. Lossless is suitable for a text document or program file because losing any characters would corrupt the file, so the data must be reconstructed exactly.
3 marksUse run-length encoding to compress the string AAAABBBCCD. Show the run pairs.
Model answer: Each run is written as (value, count): A4 B3 C2 D1. The original 10 characters are encoded as 4 run pairs. RLE is lossless because the exact original can be rebuilt by repeating each value its stated number of times.
2 marksExplain why run-length encoding would be a poor choice for the string ABABABAB.
Model answer: There are no runs of repeated characters, so each character becomes its own run pair (A1 B1 A1 B1 ...). The encoded data stores a count for every character and is therefore larger than the original, so no compression is achieved.
4 marksDescribe how a Huffman tree is built from a set of character frequencies.
Model answer: (1) Count the frequency of each character. (2) Place each character in its own node and select the two nodes with the lowest frequencies. (3) Combine them into a new parent node whose frequency is the sum of the two; add it back to the set. (4) Repeat until one node (the root) remains. Reading the tree from the root, label each left branch 0 and each right branch 1; the path to each character gives its variable-length binary code, with the most common characters getting the shortest codes.
3 marksExplain why Huffman coding produces fewer bits than fixed-length ASCII for typical text.
Model answer: ASCII uses a fixed 8 bits for every character regardless of how often it appears. Huffman coding uses variable-length codes, giving frequent characters short codes (sometimes 2 or 3 bits) and rare characters longer codes. Because the common characters dominate the message, the total number of bits is lower than 8 bits per character.
