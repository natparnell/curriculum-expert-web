How to think like a programmer (and how to debug)

The four big ideas

Sequence means the order matters: blocks run from top to bottom, one after another. Repetition (a loop) repeats the blocks inside a Repeat N times block, so you write less code. Selection uses an If … then block to make a decision: the blocks inside only run when the condition is true, for example if the path ahead is clear. A variable is a labelled box that remembers a number, like your score; Change score adds to it. Inputs are what the world tells your program (is there a star here, is a wall ahead) and outputs are what your program shows, like the Say block.

Debugging

A bug is a mistake in a program. To debug, use logical reasoning: run the program slowly, watch which block is highlighted, and compare what the robot actually does with what you expected. Press Step to run one block at a time. Press Debug to get a hint about the most likely problem. Common bugs: the robot is facing the wrong way, a loop repeats too many or too few times, or you forgot an if so the robot drives into a wall.

Why selection and a variable matter here

Each challenge is designed so a straight list of moves is not enough. You will need an if to react to the world (a wall or a star) and you will need to change the score variable when you collect things. That is the difference between a fixed set of instructions and a real program that responds to what it finds.
